# Plan: MongoDB Mongoose Schema Setup

## Overview
Convert the relational schema to MongoDB/Mongoose with transactional query support. The project currently has no database setup — this is the foundational DB layer.

---

## Step 1: Install Dependencies
```bash
npm install mongoose
```

## Step 2: Create DB Connection Utility
**File: `lib/db/connection.ts`**
- Mongoose connection singleton (cached for hot-reload in dev)
- Reads `MONGODB_URI` from environment
- Export `connectDB()` and `disconnectDB()`

## Step 3: Create Base Schema Helper
**File: `lib/db/base-schema.ts`**
- Shared fields: `created_at`, `created_by`, `updated_at`, `updated_by`, `deleted_at`, `deleted_by`, `deleted_reason`
- Export a function that spreads these fields into any schema definition

## Step 4: Create Mongoose Schemas & Models

### Status Models — all use `code` (Number) as the logical identifier + `status` (String) label
| File | Collection | Seed Data |
|------|-----------|-----------|
| `lib/db/models/department-status.ts` | `department_statuses` | `{code:1, status:"Deleted"}, {code:2, status:"Active"}` |
| `lib/db/models/role-status.ts` | `role_statuses` | `{code:1, status:"Deleted"}, {code:2, status:"Active"}` |
| `lib/db/models/page-status.ts` | `page_statuses` | `{code:1, status:"Deleted"}, {code:2, status:"Active"}` |
| `lib/db/models/permission-status.ts` | `permission_statuses` | `{code:1, status:"Deleted"}, {code:2, status:"Active"}` |
| `lib/db/models/role-permission-status.ts` | `role_permission_statuses` | `{code:1, status:"Deleted"}, {code:2, status:"Active"}` |
| `lib/db/models/user-status.ts` | `user_statuses` | `{code:1, status:"Deleted"}, {code:2, status:"Inactive"}, {code:3, status:"Active"}` |

> All status models use standard ObjectId `_id`. The `code` field is unique and indexed. Domain models reference statuses via `status_id: ObjectId` ref.

### Domain Models
| File | Collection | Key Relations |
|------|-----------|---------------|
| `lib/db/models/department.ts` | `departments` | `status_id` → DepartmentStatus, `created_by/updated_by/deleted_by` → User |
| `lib/db/models/role.ts` | `roles` | `status_id` → RoleStatus, `created_by/updated_by/deleted_by` → User |
| `lib/db/models/page.ts` | `pages` | `parent_id` → Page (self-ref), `status_id` → PageStatus, audit refs → User |
| `lib/db/models/permission.ts` | `permissions` | `status_id` → PermissionStatus, audit refs → User |
| `lib/db/models/role-permission.ts` | `role_permissions` | `role_id` → Role, `page_id` → Page, `permission_id` → Permission, `status_id` → RolePermissionStatus |

## Step 5: Create Seed Script
**File: `lib/db/seed.ts`**
- Seeds all 6 status collections (idempotent — `findOneAndUpdate` with `upsert: true` on `code`)
- Creates a default "Viewer" role with Active status (needed for user default role)
- Run via `npx tsx lib/db/seed.ts`

## Step 6: Add npm script
Add to `package.json`:
```json
"db:seed": "npx tsx lib/db/seed.ts"
```

## Step 7: Create `.env.example`
```
MONGODB_URI=mongodb://localhost:27017/it-assets
```

## Design Decisions

1. **All collections use ObjectId `_id`** — standard MongoDB convention, including status collections. A `code` field (Number) serves as the human-friendly logical identifier from the relational design.

2. **Status FK fields are `Schema.Types.ObjectId` with `ref`** — e.g., `status_id: { type: Schema.Types.ObjectId, ref: 'DepartmentStatus', required: true }`.

3. **Soft delete via `deleted_at`** — no Mongoose plugin; queries manually filter `deleted_at: null`. Explicit and simple.

4. **Transaction support** — connection utility exposes `startSession()` helper. All multi-document writes use `session.withTransaction()`.

5. **Timestamps managed manually** — `created_at`/`updated_at` set in code, not via Mongoose `timestamps: true`, for exact field control.

6. **User default role** — `role_id` references a "Viewer" role created by the seed script. Schema marks it `required: true` after seeding.
