# Plan: Make Barcode Field Non-Editable + Fix Concurrent Generation

## Current State
- Barcode is auto-generated client-side when modal opens (`generateBarcode()` via server action)
- The barcode field is editable — users can override the auto-generated value
- **Race condition**: Two users opening the modal simultaneously get the same barcode; DB unique index catches it at save time (error, no graceful retry)

## Goal
1. Make barcode read-only in the "Add New Asset" form
2. Move barcode generation to the **server side** (during `createAsset`) to eliminate the race condition entirely

## Approach: Server-Side Atomic Generation

Move barcode generation into `createAsset()` in the service layer. Use MongoDB `$inc` with an atomic counter to guarantee uniqueness even under high concurrency.

## Concurrency Answer

The **best approach for auto-generated barcodes with concurrent users** is the **atomic counter pattern** (MongoDB `findOneAndUpdate` with `$inc`). This is:
- **Atomic**: The `$inc` operation is atomic at the database level — no two reads can get the same value
- **No locks needed**: Unlike read-then-write, there's no window for race conditions
- **Simple**: No external services (Redis, UUIDs) needed
- **Sequential**: Guarantees gapless, predictable numbering (IT2600001, IT2600002, ...)

The current approach (query max barcode → increment) is vulnerable to TOCTOU (time-of-check-time-of-use) races. The atomic counter eliminates this entirely.
