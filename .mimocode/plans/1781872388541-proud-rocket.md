# Users Module - Implementation Plan

## Overview
Create the full Users CRUD module following the established entity pattern (departments/roles reference). The User model already exists at `lib/db/models/user.ts` with fields: first_name, last_name, email, password_hash, department_id, role_id, status_id, email_verified_at, is_verified, plus base audit fields.

**Design decisions:**
- Password field: **Only on create** (edit form won't include password change)
- Foreign keys: **Role (required) + Department (optional)** as dropdown selects
- Sidebar: **Skip** for now
- Soft-delete with restore capability (matching other modules)

## Key Implementation Details

### Password Handling
- On create: `password` field required in form → bcrypt hash in service
- On update: no password field → service ignores password entirely
- Never expose `password_hash` in API responses or UI

### Foreign Key Dropdowns
- Service exposes `getUserSelectOptions()` returning `{ roles: UserSelectItem[], departments: UserSelectItem[] }`
- Form modal fetches these on open via server action
- Uses existing `<Select>` UI component for dropdowns

### Soft-Delete Pattern
- Match departments module exactly: set `deleted_at`, change `status_id` to "Deleted" status
- Restore: clear `deleted_at`, set status back to "Active"

## Task: Auto-resize user form modal width based on content

### Change
In `components/modals/user-form-modal.tsx` line 119, change the `DialogContent` className:
- **Before:** `className="max-w-4xl max-h-[85vh] overflow-y-auto"`
- **After:** `className="w-fit max-h-[85vh] overflow-y-auto max-w-[90vw] sm:max-w-[70vw]"`

### Why
The current `max-w-4xl` (896px) forces a fixed width regardless of content. Using `w-fit` lets the content determine the natural width, while `max-w-[90vw] sm:max-w-[70vw]` prevents it from exceeding viewport bounds on any screen size.
