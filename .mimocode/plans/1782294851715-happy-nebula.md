# Plan: Ticket Asset Status Tracking

## Goal
When a ticket is created with an asset, store the asset's current status in the ticket (`asset_status`), then set the asset to "Repair". When the ticket is resolved or closed, restore the asset to its original status.

---

## Step 1: Add `asset_status` field to Ticket types

**File:** `lib/types/ticket.ts`

- Add `asset_status?: string` to the `Ticket` interface
- Add `asset_status?: string` to `CreateTicketInput`

---

## Step 2: Add `asset_status` field to Mongoose schema

**File:** `lib/db/models/ticket.ts`

- Add `asset_status: { type: String, default: null }` to the schema (stores the asset's original status before repair)

---

## Step 3: Update ticket service — create flow

**File:** `lib/services/ticket-service.ts` — `createTicket` function

After the ticket is created, if `asset_id` is provided:
1. Fetch the asset to get its current `status`
2. Save that status as `asset_status` on the ticket
3. Update the asset's status to `"Repair"`

---

## Step 4: Update ticket service — resolve/close flow

**File:** `lib/services/ticket-service.ts` — `updateTicket` function

When the ticket status changes to `"Resolved"` or `"Closed"`:
1. Check if the ticket has an `asset_id` and `asset_status`
2. If so, update the asset's status back to `asset_status` (the saved original status)

---

## Step 5: Update ticket `toTicket` mapper

**File:** `lib/services/ticket-service.ts` — `toTicket` function

- Include `asset_status` in the mapper output so it's available in the UI

---

## Step 6: Update ticket detail page to show asset status

**File:** `app/(dashboard)/tickets/[id]/page.tsx`

- Display the `asset_status` field (original asset status before repair) in the ticket detail view

---

## Files to modify (summary)
1. `lib/types/ticket.ts`

4. `app/(dashboard)/tickets/[id]/page.tsx`

## Verification
1. Create a new ticket with an asset attached — verify asset status changes to "Repair"
2. Check the ticket record stores the original asset status in `asset_status`
3. Resolve the ticket — verify the asset status is restored to the original value
4. Test with a closed ticket as well
5. Test creating a ticket without an asset — no status change should occur
6. Restart dev server after service changes: `Ctrl+C` then `npm run dev`
