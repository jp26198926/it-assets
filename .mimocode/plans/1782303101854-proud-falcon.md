# Fix: "Ticket not found" for non-admin users on /tickets/[id]

## Root Cause

In `lib/services/ticket-service.ts:634-655`, the `getTicketById` function applies role-based access filtering **after** Mongoose `.populate()` resolves references. After population, `requestor_id`, `created_by`, and `assigned_to` become full objects (not plain ObjectId strings). The code calls `.toString()` on these objects, which yields `"[object Object]"` instead of the actual user ID — so the comparisons against `user.userId` always fail for non-admin roles (Viewer, Technician), causing the function to return `null`.

**Key code** (`ticket-service.ts:634-655`):
```js
// After populateQuery(), these are objects, not strings:
const requestorId = ticketData.requestor_id
  ? (ticketData.requestor_id as { toString(): string }).toString()  // → "[object Object]"
  : null;
// Comparison always fails → returns null → page shows "Ticket not found"
```

Meanwhile, `getTickets` works because its role filtering is applied at the **query level** (before populate).

## Fix

**File: `lib/services/ticket-service.ts`** (lines 624-658)

Change `getTicketById` to extract IDs from populated objects correctly. The populated subdocuments have shape `{ _id: ObjectId, first_name: string, last_name: string }`. We need to access `._id.toString()` when the value is an object, or `.toString()` when it's already a string/ObjectId.

Replace the ID extraction block (lines 636-644) with logic that checks if the field is an object with an `_id` property:

```ts
function extractId(val: unknown): string | null {
  if (!val) return null;
  if (typeof val === "string") return val;
  if (typeof val === "object" && val !== null && "_id" in val) {
    return (val as { _id: { toString(): string } })._id.toString();
  }
  return (val as { toString(): string }).toString();
}
```

Then use it in the role check:
```ts
const requestorId = extractId(ticketData.requestor_id);
const createdBy = extractId(ticketData.created_by);
const assignedTo = extractId(ticketData.assigned_to);
```

## Verification

1. Log in as a **non-admin** user (Viewer or Technician)
2. Navigate to `/tickets/[id]` for a ticket you created, are assigned to, or requested — should show the ticket details (not "Ticket not found")
3. Navigate to `/tickets/[id]` for a ticket you have no relation to — should still show "Ticket not found"
4. Log in as **Administrator** — should see all tickets (no change)
