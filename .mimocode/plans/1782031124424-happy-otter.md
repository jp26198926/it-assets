# Plan: Add Cache Invalidation to Pages Service

## Context
The Pages management UI already supports editing the `path` field. However, when a page's path changes (or a page is created/deleted/restored), the authorization cache is not invalidated. This means permission checks may use stale paths for up to 5 minutes.

`invalidateCache()` exists in `lib/services/authorization-service.ts` but is never called from any page mutation.

## Changes

### File: `lib/services/page-service.ts`
1. Import `invalidateCache` from `@/lib/services/authorization-service`
2. Call `invalidateCache()` (no roleId = clears all) after each mutation:
   - `createPage()` — after the page is created
   - `updatePage()` — after the page is updated
   - `deletePage()` — after the page is soft-deleted
   - `restorePage()` — after the page is restored

Clearing the entire cache (rather than a specific role) is the correct approach because:
- Any role could reference the modified page
- Page path changes affect all roles' permission resolution
- The cache rebuilds on-demand with fresh data, so clearing is safe

### No changes needed to:
- `lib/actions/page-actions.ts` — thin wrappers, service layer handles it
- `app/api/pages/` routes — they call the service, which handles it
- `components/modals/page-form-modal.tsx` — path editing already exists
- `lib/services/authorization-service.ts` — `invalidateCache()` already works

## Verification

3. Open a new browser tab/incognito as a user with that page permission

5. Test create, delete, and restore operations to confirm cache is cleared in each case
