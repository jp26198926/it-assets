# Application Settings Page - Implementation Plan

## Overview
Create a singleton "Application Settings" page for configuring app-wide settings. Uses a **singleton pattern** (one document in MongoDB), with an inline form on the page (not a modal). No CRUD/list/delete operations needed.

## Fields

| Field | Type | Required | Default |
|-------|------|----------|---------|
| App Name | String | Yes | - |
| Tagline | String | Yes | - |
| Email | String | No | null |
| Phone | String | No | null |
| Address | String | No | null |
| TIN Number | String | No | null |
| OTP Expiry Duration | Number | No | 5 |
| Android Download Link | String | No | null |
| iOS Download Link | String | No | null |
| Facebook Link | String | No | null |
| X Link | String | No | null |
| Instagram Link | String | No | null |

Includes `BaseAuditFields` (created_at, updated_at, etc.) for consistency.

## Files to Create/Modify

### 1. `lib/types/application.ts`
- `Application` interface (all fields + audit fields, no `status_id`)
- `UpdateApplicationInput` (Partial of updatable fields only — no create input needed since singleton auto-creates)

### 2. `lib/db/models/application.ts`
- Mongoose schema with all fields
- Spread `...BaseAuditSchemaDefinition` for audit fields
- No `status_id` field (singleton has no Active/Deleted states)
- No separate status model needed
- Export: `mongoose.models.Application || mongoose.model<IApplication>(...)`
- Collection: `"applications"`

### 3. `lib/services/application-service.ts`
- `getAppSettings(): Promise<Application>` — `findOne()` on the collection; if null, create with defaults (app_name: "", tagline: "", otp_expiry_duration: 5) and return
- `updateAppSettings(data: UpdateApplicationInput): Promise<Application>` — `findOneAndUpdate({ _id: existing._id }, { $set: { ...data, updated_at: new Date() } }, { new: true })`
- No `getEntities`, `createEntity`, `deleteEntity`, `restoreEntity` — not needed for singleton
- `toApplication()` mapper for raw doc → Application type

### 4. `lib/actions/application-actions.ts`
- `"use server"` directive
- `getAppSettings()` → `applicationService.getAppSettings()`
- `updateAppSettings(data)` → `applicationService.updateAppSettings(data)`

### 5. `app/api/application/route.ts`
- `GET` — returns the singleton settings (auto-creates if missing)
- `PUT` — updates the singleton settings
- No `POST`, no `[id]` routes, no `statuses` route

### 6. `app/(dashboard)/application/page.tsx`
- Client component with inline form (no modal)
- **Layout**: Card-based sections matching the dashboard design style
  - Section 1: **App Info** — App Name, Tagline
  - Section 2: **Contact Info** — Email, Phone, Address, TIN Number
  - Section 3: **OTP Settings** — OTP Expiry Duration (number input, min 1)
  - Section 4: **Download Links** — Android Download Link, iOS Download Link
  - Section 5: **Social Links** — Facebook Link, X Link, Instagram Link
- Loads settings via `getAppSettings()` on mount
- Save button calls `updateAppSettings()`
- Uses `sonner` toast for success/error feedback
- Loading skeleton while fetching
- Form validation: App Name and Tagline required
